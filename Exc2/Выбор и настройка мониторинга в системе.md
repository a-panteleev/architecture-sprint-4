# Мотивация

Мониторинг позволит компании контролировать системные ресурсы окружений, собирать и рассчитывать метрики качества портала, вовремя реагировать на инциденты и предоставлять команде разработки ретроспективу характера нагрузки на портал.

Для бизнеса важна объективная информация по нагрузке на портал, количество обращений клиентов, количество ошибок, метрики, которые влияют на удовлетворенность пользователей.

Для команды разработки важна нагрузка и количество ошибок на каждый конкретный модуль, утилизация системных ресурсов.

Для команды поддержки \ SRE важно соответствие между согласованными и реальными метриками на каждый модуль, своевременные алерты при превышении допустимых лимитов метрик.

# Выбор подхода к мониторингу

Далее рассмотрим объекты мониторинга и подходы.

1. Виртуальные машины: имеет смысл использовать методологию USE (Utilization, Saturation, Errors), тк нам важно отследить, насколько интенсивно используются машины.
2. Managed-сервисы - базы данных и 3d file storage: тоже логично использовать USE, однако в показатели насыщенности будут уже входить интенсивность использования памяти, диска, объем трафика. Дополнительно важен request rate - количество запросов к БД.
3. Прикладные приложения - internat shop, shop api, crm, crm api, mes api, mes: я бы использовал 4 золотых сигнала (задержка, трафик, ошибки, насыщенность), тк эта методология подходит именно для систем, которые обрабатывают запросы от пользователей, в тч по API. По этим метрикам мы можем судить об удовлетворенности конечных пользователей.

# Список метрик

Ниже для каждой из предложенных метрик указано, где именно метрика будет использоваться и насколько  она актуальна.

1. Number of dead-letter-exchange letters in RabbitMQ: метрика нужна для контроля над ретрайми ивентов из очереди, я бы сказал, что для первой итерации она не нужна, только если мы не поймем, что проблема и обработкой ивентов.
2. Number of message in flight in RabbitMQ: количество сообщений, которые в данный момент найходятся в обработке со стороны consumers. Важная метрика, с ее помощью мы поймем насколько бывают загружены обработчики, возможно, нужно их скалировать.
3. Number of requests (RPS) for internet shop API: безусловно важная метрика для нагрузки на backend, добавляем в первую очередь. К данной метрике имеет смысл добавить ярлык с указанием, какой именно запрос. Так мы сможем более подробно понять характер нагрузки.
4. Number of requests (RPS) for CRM API: аналогично важно, добавляем вместе с ярлыками типа запроса.
5. Number of requests (RPS) for MES API: аналогично важно, добавляем вместе с ярлыками типа запроса.
6. Number of requests (RPS) per user for internet shop API: аналогично важно, добавляем вместе с ярлыками типа запроса.
7. Number of requests (RPS) per user for CRM API: аналогично важно, добавляем вместе с ярлыками типа запроса.
8. Number of requests (RPS) per user for MES API: аналогично важно, добавляем вместе с ярлыками типа запроса.
9. CPU % for shop API: метрика показывает утилизацию CPU от доступных значений, важно, добавляем.
10. CPU % for CRM API: аналогично важно, добавляем.
11. CPU % for MES API: аналогично важно, добавляем.
12. Memory Utilisation for shop API: показывает, насколько интенсивно приложение использует память (обычно в процентах от доступной памяти), при высоком потреблении памяти есть риск ошибок, связанных с out-of-memory, тормоза из-за запуском garbage collector, важно, добавляем в первую очередь.
13. Memory Utilisation for CRM API: аналогично важно, добавляем.
14. Memory Utilisation for MES API: аналогично важно, добавляем.
15. Memory Utilisation for shop db instance: аналогично важно, добавляем.
16. Memory Utilisation for MES db instance: аналогично важно, добавляем.
17. Number of connections for shop db instance: метрика показывает количество клиентских подключений к БД, если код написан неоптимально, происходят частые реконнекты, коннекты остаются в памяти и приводят к недоступности базы для новых клиентов. Важно, но для оперативного решения проблем, добавляем.
18. Number of connections for MES db instance: аналогично важно, добавляем.
19. Response time (latency) for shop API: показывает, насколько быстро backend отвечает на запрос, метрика нужна и важна, дополнительно можно повестить алертинг, если время обработки запроса превышает заданный лимит (например, 2 секунды), добавляем в первую очередь.
20. Response time (latency) for CRM API: аналогично важно, добавляем.
21. Response time (latency) for MES API: аналогично важно, добавляем.
22. Size of S3 storage: объем хранилища в GB, я бы сказал, это важно только если мы ставим себе ограничения. для облачного хранилища это не в первом приоритете.
23. Size of shop db instance: метрика важна, для standalone инстансов серверов БД мы можем исчерпать размеры диска, для YDB мы можем отслеживать скорость роста нашей БД с целью оптимизации. Метрика нужна и важна.
24. Size of MES db instance: аналогично важно, добавляем.
25. Number of HTTP 200 for shop API: показывает количество успешных ответов от сервера, обычно, не нужно, лучше мерить общее число запросов и отдельно число ошибок.
26. Number of HTTP 200 for CRM API: я бы отдельно не добавлял.
27. Number of HTTP 200 for MES API: я бы отдельно не добавлял.
28. Number of HTTP 500 for shop API: показывает число ошибок сервера, важная метрика, показывает, как много проблем на сервере. Важно, добавляем.
29. Number of HTTP 500 for CRM API: аналогично важно, добавляем.
30. Number of HTTP 500 for MES API: аналогично важно, добавляем.
31. Number of HTTP 500 for shop API: аналогично важно, добавляем.
32. Number of simultanious sessions for shop API: показывает количество клиентских сессий, которые обрабатываются backend. Для stateless-сервисов этот показатель не должен быть важен, важнее request rate, я бы не добавлял.
33. Number of simultanious sessions for CRM API: я бы отдельно не добавлял.
34. Number of simultanious sessions for MES API: я бы отдельно не добавлял.
35. Kb tranferred (received) for shop API: количество данных, полученных от клиента. Важно, если у нас очень дорогой трафик, много пользователей, и мы приближаемся к лимитам нашего канала. В данном случае нет указаний на эту проблему, я бы пока не добавлял.
36. Kb tranferred (received) for CRM API: я бы отдельно не добавлял.
37. Kb tranferred (received) for MES API: я бы отдельно не добавлял.
38. Kb provided (sent) for shop API: количество данных, отправленных клиенту. Исходящий канал чаще **у**же и дороже, но все равно нет никаких признаков, что мы в данном случае упираемся в лимиты нашего канала. Я бы не добавлял пока.
39. Kb provided (sent) for CRM API: я бы отдельно не добавлял.
40. Kb provided (sent) for MES API: я бы отдельно не добавлял.

# План действий
1. Добавляем дашборды для мониторинга системных ресурсов виртуалок и managed DB согласно https://yandex.cloud/en-ru/docs/monitoring/quickstart
2. Настраиваем алертинг при превышении определенного уровня метрик системных ресурсов.
3. Добавляем мониторинг для очереди согласно https://yandex.cloud/ru/docs/message-queue/metrics, добавляем алертинг.
4. Создать виртуальку для мониторинга, установить образ https://yandex.cloud/ru/marketplace/products/yc/prometheus-operator, настроить по инструкции доступ для команды поддержки.
5. Добавить агентов для сбора метрик на виртуальные машины с приложениями согласно https://yandex.cloud/ru/docs/monitoring/concepts/data-collection/unified-agent/
6. Настроить конфиги агентов для сбора системных метрик.
7. В каждое приложение добавить библиотеку для сбора метрик в формате prometheus, в конфиге агента указать url для забора метрик каждого приложения.
8. Настроить дашборды со всем объемом метрик.
9. Настроить алертинг
