# Анализ схемы и описание системы

В задании приведена архитектурная схема, краткое описание логики работы системы и список проблем, с которыми сталкиваются пользователи. Без лишнего пересказа отмечу наиболее заметные проблемы архитектуры:

1. В задании явно не указано, что используется какой-либо механизм контейнеризации и оркестрации. _Каждый из инстансов крутится как EC2_ подразумевает традиционную установку, как следствие, зависимость от железа, отсутствие встроенных средств мониторинга, отслеживание статусов и рестартов приложений.
2. Запуск каждого приложения на отдельной машине подразумевает, что нет масштабирования (ни горизонтального, ни вертикального). При этом возможна как низкая утилизация системных ресурсов в менее нагруженных приложениях, так и старвация приложений с высокой загрузкой.
3. Не увидел ни одного средства мониторинга в окружении (яндекс метрику из последующего задания не считаем), как следствие очень мало информации для отслеживания состояния системы, по сути, можно посмотреть только утилизацию самих виртуалок в EC2
4. Не увидел ни одной системы кеширования, особенно, это важно для списка заказов со статусами, дашбордов и всего необходимого для быстрой загрузки страниц.
5. Не увидел механизмов обратной связи от внутренних подсистем, в частности, неясно, как клиент получает уведомления о смене статусов, кроме как через пулинг REST API. Особенно, это было бы важно для статусов подсистемы MES.
6. Не увидел средств логирования и тем более трейсинга. Без этого сложно разбирать обращения клиентов и восстанавливать последовательность действий при необходимости.
7. Не увидел механизмов логического контроля состояния заказов для бизнеса, речь идет о какой-либо аналитической системе для отслеживания текущих заказов, контролем за статусами заказов, алертингом для службы поддержки о "зависших" заказах, длительном нахождении заказа в определенном состоянии. Необходим какой-либо внешний контроль над всей цепочкой заказов.
8. Не увидел в архитектуре средств администрирования системы со стороны команды поддержки. Неясно, как решаются проблеммные вопросы в случае обращений клиентов. Неужели приходится через разработчиков лезть в базу и править там необходимые данные?:)
9. Подозрение на сомнительное качество приобретенного софта MES - слишком долгий расчет стоимости, огромный разброс времени расчета стоимости заказа, неопримальное веб-приложение MES.

Есть проблемы и в организационной структуре компании:

1. Остутствует служба технической поддержки, неясно, кто обрабатывает жалобы клиентов через менеджеров по продажам, кто непосредственно вносит необходимые изменения в базу.
2. Остутствуют инженеры, которые обеспечивают SRE - дежурные администраторы. Для разработчиков может быть огромный стресс, если в нерабочее время происходят инциденты с приложениями в prod, которые  надо срочно чинить. А без мониторинга мы заранее не увидим риски.

Ниже предложены инициативы в порядке приоритета для решения насущных и стратегических проблем бизнеса.

# Инициативы в порядке приоритета, которые необходимы для устранения нежелательных ситуаций.

В данном разделе указаны необходимые меры в общем виде без деталей реализации, которые будут рассмотрены в последующих заданиях.
1. Вредрение инструментов мониторинга как ресурсов инстансов EC2, так и систменых ресурсов приложений.
2. Определение наиболее загруженных приложений, вертикальное масштабирование соответствующих инстансов.
3. Внедрение механизмов алертинга при превышении порога потребления ресурсов.
4. Внедрение мониторинга и алертинга для БД. Тк это managed-сервисы, будет проще настроить yandex-мониторинг.
5. Внедрение подсистемы логирования, добавление логов в код приложений в первую очередь для отслеживания изменений статусов заказов, логирование всех важных участков кода, запросов и тд.
6. Приступить к отладке наиболее медленных запросов, выявить причины - большая нагрузка на БД, неоптимальный код обработчиков, dead- или live-локи. Данную работу необходимо делать в параллель силами разработчиков, расставить приоритеты по уровню критичности для бизнеса и по сложности правок. Особенное внимание уделить приобретенному сервису MES. Добавить кеширование по результатам анализа запросов, в частности, для быстрой загрузки стартовых страниц.
7. Договориться с бизнесом о формировании команды технической поддержки. Используя инструмент логгера и простейшие запросы к БД собрать на коленке простой сервис для команды поддержки для отслеживания состояния заказов и диагностики типовых проблем. По возможности, предоставить им средства для решения типовых проблем, чтобы бизнес меньше отвлекал разработку операционкой и чтобы помогать разработке в диагностике проблем.
8. Силами devops приступить к более надежной и управляемой среде для развертывания и экплуатации приложений. Внедрить (если нет), контейнеризацию, автоскалирование, алертинг. Автоматизировать деплой в прод.
9. Договориться о метриках нагрузки для SRE, сформировать требования для SRE, потихобньку внедрять мониторинг.
10. Собираем дашборд с бизнес метриками, предоставляем инструменты для прозрачного контроля со стороны бизнеса над ключевыми метриками системы, о чем именно - договариваемся. Дорабатываем мониторинг при необходимости.
11. Внедряем инструменты трассировки для решения наиболее сложных проблем пользователей. По обращению от клиентов заводим таски и отлаживаем.
12. Постепенно у команды формируется понимание наиболее сложных участков кода и системы, наиболее серьезных проблем. По результатам анализа может быть принято решение о реинженеринге соответствующих модулей.

# Целевое видение архитектуры через полгода

Необходимо построить прозрачное с точки зрения мониторинга, решение. Основные dataflow, клиентские запросы логируются, по логам легко восстановить картину типовых ситуаций. Добавление логов для новых функций не вызывает проблем. Внедрен механизм трассировки, ключевые функции используют трассировку.
Для всех приложений используется контейнеризация, в перспективе оркестрация, все пайплайны CI\CD автоматизированы.
Решены проблемы с перфомансом наиболее проблеммных участков, по необходимости добавлен кеш, отлажен и исправлен прикладной код.
Внедрены инструменты для службы технической поддержки, разработана простейшая админка.
Внедрены средства для бизнес аналитики, начаты простейшие ETL-процессы хотя бы для мониторинга состояний заказов.

Еще более отдаленная перспектива - внедрить application monitor.

# Три наиболее важные инициативы

1. Мониторинг всех приложений, тк без отслеживания ключевых метрик системы нет понимания масштабов проблемы.
2. Масштабирование приложений - это наиболее простой, но порой необходимый способ решения проблем, вызванных старвацией системных ресурсов.
3. Добавление инструментов логирования и логов для наиболее насущных участков кода. Это важный элемент решения проблем, который поможет разработке исправить как логические ошибки, так и проблемы производительности.
